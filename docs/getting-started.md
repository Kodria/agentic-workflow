# Getting Started

The complete runbook: from "nothing installed" to "the full AWM harness running with your agent" — **from zero**.

AWM is agent-agnostic. The same workflow works for **Claude Code** and **OpenCode**; the only differences are *how each agent receives context* (Claude uses hooks, OpenCode uses config injection). Wherever the steps differ, this guide calls it out explicitly.

---

## Mental model (read this first)

AWM has **two layers**:

| Layer | Lives in | Set up by | Holds |
|---|---|---|---|
| **Machine (global)** | `~/.awm/`, plus the agent's global skills dir | once per machine | the CLI cache, the baseline skill pack, the agent's context-delivery mechanism |
| **Project (per repo)** | `<repo>/.awm/`, `CONSTITUTION.md`, sensor configs | once per repo | the project profile, sensor manifest, project rules |

And **one command bootstraps both**: `awm init`. You almost never run the low-level commands (`awm hooks install`, `awm sensors init`, …) by hand anymore — `init` orchestrates them and is **idempotent**, so you can re-run it any time. `awm doctor` shows you the state of both layers without changing anything.

**Where each agent stores skills:**

| Agent | Global skills | Context delivery |
|---|---|---|
| Claude Code | `~/.claude/skills/` | `SessionStart` hook injects `using-awm` + `CONSTITUTION.md` |
| OpenCode | `~/.agents/skills/` | `~/.config/opencode/opencode.json` `instructions[]` (global AWM context) + `<repo>/opencode.json` `instructions[]` (project `CONSTITUTION.md`) |

---

## Prerequisites

- `git`, `node`, `npm` on your `$PATH`
- A working directory inside a git repo where you want the system active

---

## 1. Install AWM (once per machine)

```bash
curl -fsSL https://raw.githubusercontent.com/Kodria/agentic-workflow/main/install.sh | bash
awm --help        # verify it's on PATH
```

The installer clones the registry into `~/.awm/cli-source` and links the `awm` binary. Nothing is wired into any agent yet — that's the next step.

---

## 2. Bootstrap everything with `awm init`

Run `awm init` **inside the repo** where you want the harness. Pick your agent:

```bash
# Claude Code (the default)
awm init

# OpenCode
awm init --agent opencode
```

> `awm init` is idempotent — re-run it any time. To bootstrap only the machine layer (no project changes), use `awm init --machine-only`. Add `--yes` to skip prompts (for scripts).

**What `awm init` does in one pass:**

- **Machine:** syncs the registry cache, installs the agent's context mechanism (Claude: `SessionStart` hook; OpenCode: `instructions[]` in the global `opencode.json`), and installs the **baseline skill pack** (`dev`) — the full spine: `using-awm`, `development-process`, `brainstorming`, `writing-plans`, `subagent-driven-development`, `test-driven-development`, `systematic-debugging`, `verification-before-completion`, `post-implementation-qa`, `harness-retro`, `setup-sensors`, `project-constitution`, and more.
- **Project:** bootstraps `.awm/profile.json`, detects your stack and writes `.awm/sensors.json` (+ copies the pack's config files), and — for OpenCode — wires `CONSTITUTION.md` into the repo-local `opencode.json` when one exists.

**What it leaves for you** (these need an agent or a deliberate choice — `init` flags them, it doesn't do them):

- The **PostToolUse sensor hook** (Claude only — auto-runs fast sensors after each edit): `awm sensors install`
- **`CONSTITUTION.md`** — generated by the `project-constitution` skill (Section 4)
- **Agent context** (`CLAUDE.md` / `AGENTS.md`) — generated by the `project-context-init` skill (Section 4)

### Read the state with `awm doctor`

```bash
awm doctor
```

```
AWM · estado del harness

Máquina (global)
  ✔ CLI v1.0.0
  ✔ hook SessionStart
  ✔ dev-core (baseline)
  ✔ skills globales

Proyecto: my-repo
  ✔ .awm/profile.json (sin extensiones)
  ✔ bundles activos
  ✔ sensores
  ✖ CONSTITUTION.md ausente        → skill: project-constitution
  ⚠ contexto del agente ausente    → skill: project-context-init
```

Read the glyphs: `✔` healthy · `⚠` advisory (won't block) · `✖` missing (degrades state). Each `✖`/`⚠` carries its own remedy — either a command (`→ awm …`) or a skill to ask your agent to run (`→ skill: …`). A row that says `→ skill: project-constitution` is **not a bug**: it's `init` telling you the next step requires the agent. The state stays `degradado` until you run those skills — that's expected on a fresh repo.

## 3. Let the agent receive AWM (first session)

The whole point is that **every agent session starts already knowing the AWM discipline**. Trigger a fresh context load:

- **Claude Code:** restart, or run `/clear`. The `SessionStart` hook injects `using-awm` (+ `CONSTITUTION.md` once it exists) into the initial context.
- **OpenCode:** start a new session. The `instructions[]` entries in `opencode.json` (global AWM context + project `CONSTITUTION.md`) load each session.

To confirm it worked, ask the agent something like *"¿qué skills de AWM tenés disponibles?"* — it should reference `development-process` and the spine skills.

---

## 4. Finish project setup (the skills `doctor` flagged)

These run **through your agent**, in the session you just started. Ask in plain language; the agent invokes the skill.

### 4.1 Generate the project constitution

`CONSTITUTION.md` holds the repo's **non-negotiable rules**. AWM delivers it into every session (Claude: hook; OpenCode: repo-local `opencode.json`).

> *"Generá la `CONSTITUTION.md` con `project-constitution`."*

The skill reads `CLAUDE.md`/`AGENTS.md`/`README`/`.awm/sensors.json`, drafts the rules section by section (with your approval), writes the file at the repo root, and commits it. After it exists, re-trigger a session load (Section 3) so the agent starts receiving it.

### 4.2 Generate the agent context file

`AGENTS.md` (agent-agnostic) or `CLAUDE.md` (Claude-specific) **describes** the repo — purpose, structure, commands. (Rules go in `CONSTITUTION.md`; description goes here.)

> *"Inicializá el contexto del proyecto con `project-context-init`."*

Prefer `AGENTS.md` — every agent reads it. Use `CLAUDE.md` only for Claude-specific notes.

### 4.3 Sensors: adapt and (optionally) baseline

`awm init` already wrote `.awm/sensors.json` and copied the pack configs. Check their health:

```bash
awm sensors status
```

- `HEALTHY` — ready.
- `DEGRADED` — the templated config doesn't match your installed tool versions. Ask the agent: *"Adaptá los sensors con `setup-sensors`."*
- Tools that run via `npx` (eslint, depcruise) must be **devDependencies**, or `npx` fetches them remotely and `status` reports them as not installed. Fix with `npm i -D <tool>`.

On a **legacy repo** a sensor may report thousands of pre-existing findings. Snapshot them as an accepted baseline so sensors fail only on **new** findings, then commit it:

```bash
awm sensors baseline      # writes .awm/sensors.baseline.json — commit it
```

Skip the baseline on a clean/greenfield repo.

### 4.4 (Claude only) Install the per-edit sensor hook

```bash
awm sensors install
```

This installs the `PostToolUse` hook so **fast** sensors (tsc/eslint) run automatically after each file edit. **OpenCode has no hooks** — it runs sensors at the completion gate instead (the agent invokes `awm sensors run`, driven by `verification-before-completion`).

## 5. What happens automatically, day to day

| Trigger | Claude Code | OpenCode |
|---|---|---|
| New session in the repo | `SessionStart` hook injects `using-awm` + `CONSTITUTION.md` | `instructions[]` load the global AWM context + project `CONSTITUTION.md` |
| Agent edits a file | `PostToolUse` hook runs fast sensors, surfaces findings | — (no per-edit hook) |
| Agent about to declare "done" | `verification-before-completion` runs `awm sensors run` and reads the output | same |
| Same sensor fails a 2nd time | the skill recommends `harness-retro` to turn it into a rule | same |

The common thread: **recurrence becomes a rule, not a repeated symptom fix.**

## 6. The development lifecycle

Your default entry point for any development work is the **`development-process`** orchestrator. It reads project state and routes between phases — you don't pick the skill, it does:

```
brainstorming → writing-plans → subagent-driven-development → post-implementation-qa → finishing-a-development-branch
```

Just describe the work — *"Quiero agregar la feature X"* — and let `development-process` drive. Cross-cutting skills (`test-driven-development`, `systematic-debugging`, `verification-before-completion`) kick in within phases.

### The learning loop (ledger + harness-retro)

AWM remembers what goes wrong **and** what goes right, per branch, without bloating your context:

- During development, skills append wins and findings to a per-branch **ledger** (`awm ledger`) — ephemeral working memory, gitignored, never injected into the agent's context.
- At the end of a branch, **`harness-retro`** reads the ledger and *cures* recurring lessons into your durable docs: structural findings → the remediation tree (eslint/semgrep/structural tests), process lessons → `CONSTITUTION.md`, agent lessons + wins → `AGENTS.md`. Then it archives the ledger.

You rarely touch `awm ledger` directly — the skills do. Inspect it if curious: `awm ledger list` / `awm ledger recurring --min 2`.

> **Project-specific vs framework rules.** Rules born from a bug in *your* repo live in *your* repo (grown by `harness-retro` into your config files / `CONSTITUTION.md`). Only universally-evitable patterns (e.g. "never `eval`") belong in the AWM registry. AWM ships the *mechanism*, not your project's bug list.

## 7. Keep everything up to date

```bash
awm update
```

This pulls the latest registry **and rebuilds the CLI binary** (you do not run `npm build`). Because skills are symlinked into the cache by default, the update instantly patches every global and local install on the machine. Re-run `awm init` afterward if you want the machine/project wiring reconciled to any new defaults (it's idempotent).

## 8. Command reference (the surface you actually use)

| Command | What it does |
|---|---|
| `awm init [--agent <a>] [--machine-only] [--yes]` | Bootstrap machine + project (idempotent). `<a>` = `claude-code` (default) / `opencode` / `antigravity` |
| `awm doctor [--json]` | Read-only dashboard of machine + project state |
| `awm sensors status \| run \| init \| install \| baseline` | Manage computational sensors for the current project |
| `awm add [name]` / `awm list` / `awm remove` | Install / browse / uninstall individual skills, workflows, processes |
| `awm sync` | Rebuild local skill symlinks from `.awm/profile.json` (e.g. after cloning on a new machine) |
| `awm update` | Sync the registry and rebuild the CLI |
| `awm hooks status \| install \| uninstall` | Manage the Claude `SessionStart` bootstrap hook |
| `awm ledger list \| recurring \| add \| archive` | Per-branch findings ledger (mostly driven by skills) |

## 9. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `awm: command not found` | binary symlink points at a missing `~/.awm/cli-source/cli` | re-run the install script |
| `awm doctor` shows `✖ .awm/profile.json → awm init` right **after** running `awm init` | (fixed) profile wasn't bootstrapped on zero-extension repos | `awm update` to rebuild the CLI, then `awm init` |
| New session doesn't show `using-awm` in context | session opened before the wiring existed | Claude: `/clear` or restart · OpenCode: start a new session |
| Sensors never run after an edit (Claude) | `PostToolUse` hook not installed | `awm sensors install` |
| Sensors don't run for OpenCode after edits | by design — OpenCode has no hooks | they run at the completion gate via `awm sensors run` |
| `awm sensors status` says a tool is "not installed locally" | `npx` tool missing from devDependencies | `npm i -D <tool>` |
| Sensor always red on a legacy repo | no baseline accepted yet | `awm sensors baseline` (Section 4.3) |
| `DEGRADED` sensor that won't clear | config templated for a different tool version | ask the agent to run `setup-sensors` |
| A skill isn't in the agent's catalog | baseline pack not installed | re-run `awm init` (installs the `dev` baseline) |
| OpenCode isn't receiving `CONSTITUTION.md` | repo-local `opencode.json` missing the entry | re-run `awm init --agent opencode` (it wires it when `CONSTITUTION.md` exists) |
